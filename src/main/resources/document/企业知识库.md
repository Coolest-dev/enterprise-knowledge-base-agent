## 第一章：企业文化与核心价值 (Corporate Culture & Core Values)

### 1.1 企业使命与愿景
* **使命 (Mission)**: “用技术连接未来，以创新驱动变革。” 我们致力于通过前沿的云计算、人工智能和隐私安全计算技术，为全球企业提供数字化转型底层基建。
* **愿景 (Vision)**: 成为全球最值得信赖的分布式科技与数据安全解决方案服务商。

### 1.2 核心价值观 (Core Values)
1. **技术立身 (Tech-First)**: 鼓励深钻技术底层，拒绝浮躁，用高质量的代码和严谨的架构解决实际问题。
2. **信任与透明 (Trust & Transparency)**: 内部信息高效共享，倡导基于事实的理性讨论，推崇“对事不对人”的 Code Review 文化。
3. **客户成就 (Customer Success)**: 深入理解客户业务痛点，绝不为了技术而技术，交付能够真正带来商业价值的产品。
4. **包容与自驱 (Inclusivity & Self-Driven)**: 支持弹性工作与远程协作，信任员工的自我驱动力，提供多元平等的职业发展通道。

## 第二章：组织架构与全流程协作 (Organizational Structure & Collaboration)

### 2.1 核心部门职责一览
* **产品研发部 (R&D Department)**: 负责核心 SaaS 平台、LDP（本地差分隐私）数据安全中台、AI 算法模型的开发与维护。
* **产品管理部 (Product Management)**: 负责市场调研、PRD 撰写、产品生命周期管理以及跨部门需求排期。
* **全球交付与服务部 (Global Delivery & Support)**: 负责大客户定制化部署、售后技术支持（L2/L3）及 SLA 保障。
* **市场与销售部 (Marketing & Sales)**: 负责海内外 B 端市场拓展、品牌建设、KA 客户线索跟进。
* **职能支持部 (Operations & HR)**: 负责人才招聘、绩效考核（OKR）、财务审计与法务合规。

### 2.2 跨部门协作标准工作流 (Workflow)
```
[客户需求/市场洞察] ➔ [产品部: PRD评审] ➔ [研发部: 架构设计与评审] ➔ [双周迭代开发]
                                                                        │
[客户交付/上线] ⮀ [运维部: 自动化部署/CI-CD] ⮀ [QA部: 自动化+性能测试] ⮀ [Code Review]
```

1. **需求提出阶段**: 销售或市场部通过 Jira 提交 `Feature Request`，产品经理（PM）在 3 个工作日内完成初步评估。
2. **设计与评审**: 涉及重大功能变更时，需由架构师组织跨部门技术评审，并输出 `RFC (Request for Comments)` 文档。
3. **Sprint 迭代**: 研发团队采用标准敏捷开发（Scrum），双周为一个迭代周期。每周一上午举行 15 分钟站会（Stand-up Meeting）。

## 第三章：技术栈与开发规范 (Tech Stack & Development Guidelines)

### 3.1 核心技术栈 (Technology Stack)
* **后端支撑**: Java (Spring Boot 3.x / Cloud), Python (FastAPI / PyTorch 用于数据挖掘与隐私计算)
* **前端架构**: TypeScript, React 18, Next.js, Tailwind CSS
* **数据存储**: PostgreSQL (关系型), Redis (缓存), ClickHouse (大容量分析)
* **基础设施**: AWS / 阿里云, Kubernetes (K8s), Docker, GitHub Actions (CI/CD)

### 3.2 代码管理规范 (Git Branching Model)
团队严格遵循 `Git Flow` 简化版模型：
* `main`: 生产分支，仅接收来自 `release` 或 `hotfix` 的合并，严禁直接提交。
* `release`: 预发布分支，用于部署 UAT 测试环境。
* `develop`: 主开发分支，所有功能分支的集散地。
* `feature/*`: 特征开发分支，从 `develop` 拉取，命名格式为 `feature/jira-ID-description`。
* `hotfix/*`: 紧急修复分支，从 `main` 拉取，修复后需同时合并回 `main` 和 `develop`。

#### Commit Message 提交规范
必须符合 Angular 规范，格式为：`<type>(<scope>): <subject>`
* `feat`: 新增功能
* `fix`: 修复 Bug
* `docs`: 文档变更
* `style`: 代码格式调整（不影响逻辑）
* `refactor`: 重构代码
* `test`: 增加或修改测试用例

*示例*: `feat(auth): add OAuth2 login support for enterprise clients (#102)`

### 3.3 后端多线程与高性能规范 (以 Python 为例)
在处理海量数据或密集型计算（如隐私保护数据挖掘、图算法）时，为突破 Python GIL 限制，必须遵循以下多进程（Multiprocessing）编写规范：
1. **避免在子进程中共享大对象**: 应使用 `Initializer` 初始化只读全局变量，或利用 `multiprocessing.shared_memory` 降低内存开销。
2. **正确关闭进程池**: 必须使用 `with` 语句或显式调用 `pool.close()` 与 `pool.join()`，防止孤儿进程占用系统 CPU 资源。
3. **异常捕获**: 子进程内的逻辑必须包含完整的 `try-except` 块，并将错误日志上报至主进程，避免子进程静默退出导致主进程无限死锁。

## 第四章：企业信息安全与合规 (Information Security & Compliance)

### 4.1 数据安全基本原则
NexTech Solutions 处理大量涉及企业客户及用户隐私的数据，安全是我们的生命线：
* **最小权限原则 (Least Privilege)**: 员工仅拥有完成当前工作所需的最低系统权限。生产环境权限需通过堡垒机临时申请。
* **零信任架构 (Zero Trust)**: 无论是在办公室还是远程办公，访问内部系统均需挂载公司专用 VPN 并通过多因素认证（MFA）。
* **严禁泄露源码与机密**: 严禁将公司代码、客户数据、配置文件上传至个人 GitHub、网盘或公开的 AI 聊天工具中。

### 4.2 隐私计算与合规标准 (LDP/GDPR)
1. **本地差分隐私 (Local Differential Privacy, LDP)**: 在收集用户端行为数据时，必须在客户端注入满足 $\epsilon$-差分隐私的噪声（如使用 Randomized Response 或 Harmony 机制），确保服务器端无法逆向推导单个用户的真实属性。
2. **合规性审计**: 产品设计须严格遵守欧盟 `GDPR`、中国 `《数据安全法》` 及 `《个人信息保护法》`。敏感信息在存储时必须进行非对称加密（AES-256）与脱敏处理。

## 第五章：日常办公与行政人事指南 (Operations & HR Guide)

### 5.1 工作时间与弹性考勤
* **标准工作时间**: 周一至周五 09:30 - 18:30（包含 1 小时午休）。
* **弹性机制**: 核心协作时间为 10:30 - 16:30，员工可在该区间前后弹性调整上下班时间。
* **远程办公 (WFH)**: 每位全职员工每周享有最多 2 天的远程办公额度，需提前在飞书/钉钉系统上提交“远程办公申请”并知会直属主管。

### 5.2 常用系统与IT工具配置
* **即时通讯 & 办公自动化**: 飞书 (Lark) — 用于聊天、日历同步、云文档协作、审批流。
* **项目管理**: Jira / Confluence — 用于管理 Issue、Bug 追踪及长期技术知识沉淀。
* **代码托管**: GitHub Enterprise — 公司私有代码仓库。

### 5.3 报销与差旅标准
1. **因公差旅**: 需提前 3 个工作日提交《差旅申请单》。机票默认经济舱，高铁默认二等座。
2. **住宿标准**: 一线城市（北京/上海/深圳/东京）不超过 500 元/晚，二线城市不超过 350 元/晚。
3. **报销时效**: 发生费用后的 30 天内，需将电子发票上传至财务报销系统，并将纸质发票贴好交至财务部。

## 第七章：应急响应与故障排查 (Incident Response)

### 7.1 生产环境故障等级定义
* **P0 (致命故障)**: 核心服务大面积不可用（如 SaaS 平台整体宕机、数据发生泄露、支付系统瘫痪）。要求：5分钟内响应，架构师及以上全员上线。
* **P1 (严重故障)**: 部分核心功能受损，影响大量用户（如某个主要 API 持续报错 500，导致部分区域无法登录）。要求：15分钟内响应。
* **P2 (一般故障)**: 非核心功能异常，存在替代方案（如后台管理系统的导出报表功能慢、UI 样式错位）。要求：2小时内响应。
